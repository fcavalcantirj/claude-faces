=== 2026-07-18 — Implement Web Speech API TTS with chunking and boundary-driven mouth [frontend] ===
Added lib/tts/web-speech.ts — the zero-infra voice-out engine wrapping the browser SpeechSynthesis
API into the same shape the (later) TTS router drives: speak(text), cancel(), and lifecycle
callbacks (onStart/onEnd/onBoundary/onError/onMouth/onVoicesChanged). Two Web-Speech realities are
worked around and documented in a load-bearing header comment: (1) the ~15s Chrome truncation bug —

each well under the window), each <= MAX_UTTERANCE_CHARS=200; an over-long sentence is split on word
boundaries (splitLong) and a monster word is hard-split. Sentence splitting REUSES the chat client's
tested splitSentences (append '\n' to force the final terminator to flush). Utterances are spoken
ONE AT A TIME — the next chunk is only handed to the synth on the previous utterance's onend — so the
synth is never over-queued and cancel() stops instantly. (2) SpeechSynthesis exposes no tappable
AudioNode, so the mouth is driven by a SYNTHETIC envelope: an internal rAF loop calls
estimateFeatures(activeUtteranceText, now - utteranceStartedAt) (from lib/lipsync) each frame and
writes {open, viseme} into the shared mouthRef (mouthSource 'estimated') + fires onMouth; onboundary
word events are forwarded (BoundaryInfo{charIndex,name,utterance}) so the orchestrator can track
progress. Emotion is left to callbacks per the seam pattern: onStart fires ONCE on the first
utterance's start (-> orchestrator sets 'speaking') and onEnd fires ONCE after the final utterance's
onend (-> return to the resting emotion). Barge-in: cancel() bumps an internal generation counter so
any late onstart/onend/onerror from the cancelled utterance no-ops (never advances the queue or fires
onEnd), clears the queue, stops the frame loop, silences the mouth, and calls synth.cancel() — but
ONLY when speech was actually active, so the pre-speak reset speak() does on an idle controller is not

ROOT CAUSE, and it is a nasty one: dev.mjs found the port holder with `lsof -ti tcp:PORT` and,
when lsof was missing, returned [] — silently reporting "the port is free" and skipping the
kill. The original comment even said "lsof missing (e.g. Windows) — treat as free". A stock
node:22-bookworm image has NONE of lsof, ss, fuser OR netstat (verified), so on Linux that was
not an edge case, it was the NORMAL case. The project's hard requirement — always kill any
previous dev server before starting — silently did nothing on Linux, and reported success.
FIX: a strategy chain, still strictly PORT-SCOPED (never kill by process name, which would nuke
unrelated projects' servers): lsof -> pure-Node /proc scan -> ss/fuser -> netstat on Windows.
The /proc path is the one that matters for containers and needs no external binary at all: parse
/proc/net/tcp and tcp6 for a LISTEN (state 0A) socket on the port, take its inode, then walk
/proc/<pid>/fd/ looking for a symlink to socket:[inode]. When NOTHING can inspect the port it
now WARNS explicitly instead of returning [] — a quiet wrong answer is worse than a loud unknown,
which is exactly what the original bug proved.
THE TEST HAD THE SAME BUG. dev.test.ts's portOwners() helper also shelled out to `lsof` directly,
so on Linux its own SANITY CHECK ("the decoy really is holding the port") failed with "expected 0
to be greater than 0" BEFORE the real assertion ever ran. The test was macOS-only while claiming
to verify portable behaviour — a parallel imitation of the code under test, carrying the same
defect. Now imports findPidsOnPort from dev.mjs so it exercises the SHIPPED lookup.

artifacts of MY OWN tooling — macOS BSD tar wrote AppleDouble ._* resource-fork files into the

2026-07-19: [frontend] DETERMINISTIC SPEECH FLUSH — Felipe's second live report on the mute-

stream loop. After each delta, one timer is armed: buffer ends with a terminator ->

boundary with the trailing-newline trick chunkForSpeech already uses, never speaks a half-open
[[face: directive, clears on done/error/abort, and empties the buffer so the done flush cannot
re-speak. Anything the agent says is now SPOKEN within ~1.5s of arriving, no matter what
